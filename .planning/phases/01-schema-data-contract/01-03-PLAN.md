<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| spam layer process → contract views | downstream consumer reads concurrently via a separate connection/role |
| caller → upsert_pubkey | 32-byte pubkey input crosses into storage; length validated at the store boundary |
| caller → apply_follow_list | followee id list (untrusted size in later phases) crosses into the writer transaction |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | SQL injection in store queries | mitigate | All store queries use sqlx `query!`/`query_scalar!`/`query_as!` parameterized macros; never string-format SQL (Security Domain). |
| T-03-02 | Tampering / DoS | malformed/oversized pubkey input | mitigate | `upsert_pubkey` enforces exactly 32-byte length at the boundary (V5); rejects otherwise with StoreError. Follow-list cap is INGEST-04 (Phase 2); the schema/writer do not assume bounded input. |
| T-03-03 | Elevation of Privilege | over-privileged consumer access | mitigate (documented) | SCHEMA.md prescribes a read-only role granting SELECT on the three contract views only (V4); enforcement is operator DB config. |
| T-03-04 | Information Disclosure | DB URL leakage in logs | mitigate | DB URL loaded via `config`/env; never logged; kept out of git (Security Domain). |
| T-03-05 | Tampering | non-atomic edge diff leaves half-applied state | mitigate | Entire diff runs in one `pool.begin()`/`tx.commit()` transaction (Pitfall 4); `edge_diff` tests assert consistency. |
</threat_model>
