<objective>
Close WR-03 (BLOCKER) — RELAY-02 and RELAY-04 mechanisms are library code with
passing unit tests but ZERO callers in the production path. `acquire()`,
`LimitCache::get_or_fetch()`, and `record_notice()` are never reached from
`acquire_validated_lists_client` / `fetch_complete`, so the rate limiter, the
NIP-11 cache, and notice-driven backoff do not actually govern any real REQ. This
makes "politely" and the RELAY-02 cap unenforced in production.

This plan WIRES the corrected mechanisms (delivered by 02-05 fetch, 02-07 nip11,
02-08 rate_limit) into the production acquisition path: gate every `fetch_events`
behind `acquire(relay_url)`, source `max_limit` from `LimitCache`, and spawn a
`notifications()` consumer feeding `record_notice()`/`backoff()`.

Purpose: turn RELAY-02/RELAY-04 from tested-but-disconnected library code into
enforced production behavior, restoring the data-flow links the verification
flagged DISCONNECTED.

Output: a wired `acquire_validated_lists_client` and `fetch_complete` path plus a
test that proves the mechanisms are reached on the production seam.
</objective>
