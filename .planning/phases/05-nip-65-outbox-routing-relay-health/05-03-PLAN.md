<objective>
Wire the RELAY-05 fallback into `process_batch`: at the `not_found` (None) arm, resolve the pubkey's NIP-65 write relays (on-demand curated kind:10002 fetch + persist if unknown), prefer healthier relays capped at `nip65_max_write_relays`, fetch kind-3 from them via an injected `fallback_fetch` closure, re-resolve through the existing single-author `acquire_validated_lists` pass, and either `apply_validated` (recovered, counter++) or stamp terminal `not_found`. The crawl layer stays free of the live `Client` (closure injection, ScriptedGraph-testable).

CONTEXT requires "persist relay lists whenever a kind:10002 winner is seen." In this phase the daemon fan-out only ever fetches kind-3 — it never passively fetches kind:10002 — so the ONLY path on which a kind:10002 winner is seen is the on-demand curated fetch at this None (not_found) arm. That on-demand fetch persists the winning relay list via `apply_relay_list` before it is consumed. Therefore "persist whenever a kind:10002 winner is seen" is fully satisfied by this single hook; no separate passive-ingest task is needed and no kind:10002 winner is observed-but-unpersisted anywhere in the phase.

Purpose: This is the RELAY-05 recovery behavior — pubkeys the curated set misses are recovered from their advertised write relays. It depends on 05-01 (store/extraction) and 05-02 (health score for relay ordering).
Output: extended `process_batch` (new injected `fallback_fetch` closure + fallback config params + health-score ordering), the on-demand-resolve helper, and green `tests/nip65_fallback.rs` recovery + miss tests. Regenerated `.sqlx/` if any new query macros are added.
</objective>
