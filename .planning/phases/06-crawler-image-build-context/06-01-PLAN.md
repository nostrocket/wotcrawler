<objective>
Deliver the three committed build-context artifacts that let an operator build a small, secure, runnable `crawler` image from source: a multi-stage `Dockerfile`, a `.dockerignore`, and the `.env` addition to `.gitignore`.

Purpose: Satisfy IMAGE-01 (multi-stage build, no live DATABASE_URL), IMAGE-02 (non-root runtime, no build toolchain in final image), IMAGE-03 (secret-free build context). Seeds the CONFIG-02 gitignored-`.env` contract that Phase 7 completes.

Output: `Dockerfile` (4-FROM multi-stage cargo-chef build), `.dockerignore`, modified `.gitignore`.

Scope discipline: This plan delivers ONLY these three files. It does NOT touch runtime config supply, `--config`/`WOT__*` wiring, compose, healthchecks, port publishing, `.env.example`, graceful-drain, or "Run with Docker" docs — all Phase 7 and explicitly out of scope. The default `ENTRYPOINT ["/crawler"]` makes the image buildable + runnable-with-args, not self-starting.

Claude's-Discretion decisions made for this plan:
- Stage names: `chef`, `planner`, `builder`; runtime stage unnamed. Matches the CONTEXT.md endorsed skeleton.
- No BuildKit `--mount=type=cache`: cargo-chef already provides the dependency-layer caching D-05 requires; a cache mount adds BuildKit-version coupling for marginal gain. Kept off to match the endorsed skeleton exactly.
- ENTRYPOINT: exec form `["/crawler"]` (D-Discretion default).
- EXPOSE: add `EXPOSE 9100` as a documentation-only hint for the metrics port; this publishes nothing (port publishing remains Phase 7).
</objective>

    Stage 1 `chef`: base `rust:1.94-bookworm` (D-04, tag-pinned, no @sha256 per D-10); install cargo-chef.
    Stage 2 `planner` (FROM chef): `COPY . .` then `cargo chef prepare --recipe-path recipe.json` (D-05).
    Stage 3 `builder` (FROM chef): copy `recipe.json` from planner; `cargo chef cook --release` (cached dependency layer for ~200 crates); then `COPY . .` and run the release build with `SQLX_OFFLINE=true` set (D-06, IMAGE-01) — `cargo build --release` against the committed `.sqlx/`, NO live DATABASE_URL.     Stage 4 runtime (FROM `gcr.io/distroless/cc-debian12:nonroot`, D-01, tag-pinned no digest D-10): `COPY --from=builder /app/target/release/crawler /crawler` (ONLY the binary, D-07 — do NOT copy `migrations/`, `.sqlx/`, or source). Set `USER nonroot` (built-in uid:gid 65532:65532, D-08 — no useradd). `EXPOSE 9100` as a doc-only metrics-port hint (Claude's Discretion). `ENTRYPOINT ["/crawler"]` exec form (D-Discretion default; image is runnable-with-args, not self-starting — config supply is Phase 7).
    Confirm the working directory the binary path assumes: cargo-chef/cargo default WORKDIR in `rust:*-bookworm` is `/app` only if set; the skeleton's `/app/target/release/crawler` path implies a `WORKDIR /app`. Set `WORKDIR /app` in the `chef` stage so `COPY . .`, `recipe.json`, and the `target/release/crawler` artifact path are all under `/app` and the runtime COPY source path resolves.
    Do NOT add a digest pin, a healthcheck, a CMD config path, or any runtime config/env wiring — those are Phase 7.
    Read-only-rootfs posture (D-09): the binary writes nothing to local disk; no tmpfs declared. This is a property of the image, not a Dockerfile directive — no action needed beyond not introducing local writes.

       `docker inspect --format '{{.Config.User}}' wot-crawler:phase6`

       `docker run --rm wot-crawler:phase6 --help`
       Expected: the crawler prints its clap usage/help (it requires `--config`; `--help` short-circuits before config load) and exits. This proves the binary is present, executable, and dynamically links against the distroless glibc + ca-certificates — no missing-shared-library error.

Optionally confirm `.dockerignore` is respected by checking the "transferring context" size is small.
