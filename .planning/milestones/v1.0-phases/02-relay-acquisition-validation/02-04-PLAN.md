<objective>
Wire the two halves of Phase 2 together: connect the relay acquisition output (the raw, deduplicated event stream produced by `src/relay/fetch.rs`'s `fetch_kind3_complete` pagination loop) through the ingest orchestrator in `src/ingest/mod.rs` (verify gate -> dedup seen-set -> replaceable resolution -> follow-list bounds) so that `ValidatedFollowList` values actually emerge from the acquisition half. Plans 02-02 and 02-03 build the two halves in parallel but leave them unconnected; this plan builds the `acquire_validated_lists` seam in `src/relay/mod.rs` and proves it end-to-end against the plan-02-03 mock relay.

Purpose: The phase goal is that "only correct, deduplicated, newest-wins follow lists emerge" — that is only observable when fetch -> ingest is wired. Two correct-but-unconnected halves do NOT satisfy the goal (RESEARCH Anti-Pattern: "Building the two halves but never connecting them"). This plan closes the checker's key_links_planned warning by making the relay->ingest seam a first-class, tested artifact (RESEARCH Architectural Responsibility Map: "fetch->ingest pipeline wiring … wired by plan 02-04").

Output: A wired `acquire_validated_lists` pipeline function in `src/relay/mod.rs` and a green end-to-end test `tests/acquire_pipeline.rs` asserting a `ValidatedFollowList` emerges with the correct deduped, newest-wins, self-drop-filtered followee set.
</objective>

|----------|-------------|
| relay fetch stream -> ingest gate | This plan creates the actual data path across the project's primary input-validation boundary (ASVS V5). A wiring bug here would let unverified relay events bypass the gate and reach a ValidatedFollowList — the highest-severity failure mode in Phase 2. |
| paged windows -> single resolution pass | Events arrive across multiple paged windows; the seam must hand the full union to one resolution pass so a relay cannot split a pubkey's events across windows to defeat newest-wins. |
