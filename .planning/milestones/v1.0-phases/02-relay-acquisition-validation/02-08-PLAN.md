<objective>
Close CR-05 (BLOCKER) and WR-01 (WARNING) in the rate limiter — the "politely"
half of the phase goal. Today `acquire()` REMOVES the limiter from the map before
awaiting; a concurrent caller finds the slot vacant and mints a fresh full-burst
limiter, so quota is multiplied by concurrency (T-02-10 politeness void). On
return `or_insert(limiter)` keeps the wrong (stateless) entry, discarding accrued
GCRA state. Separately, `backoff_delay_unjittered` uses `u128::checked_shl`, which
returns `Some(0)` for failure counts 119-127 at base=1s (high bits truncated) — so
after 119+ consecutive failures the crawler retries with ZERO delay, the opposite
of saturation.

NOTE: production-path WIRING of acquire()/record_notice()/LimitCache is CR-05's
WR-03 component and is handled in plan 02-09 (it depends on this plan landing
first). This plan fixes the limiter's internal correctness only.

Purpose: make per-relay throttling correct under concurrency and the backoff
schedule monotonic-to-cap, so RELAY-04 politeness holds.

Output: `Arc<DirectLimiter>` storage with clone-and-await acquire(), an early
saturation guard in `backoff_delay_unjittered`, and tests for both.
</objective>
