<objective>
Close the single remaining BLOCKER gap (CR-01-new, RELAY-03) from
02-VERIFICATION.md: the no-newer-event boundary stall in `paginate_chunk`
(`src/relay/fetch.rs`). Plan 02-10 added a `prev_until`-based stall guard that
fires only on the SECOND visit to a pinned `until=T`. When ALL pool events share
the boundary second `T` (no event newer than `T`), `prev_until` is `Some(now)`
when `current_until` first becomes `T`, so the comparison
`prev_until == Some(current_until)` is FALSE on the second window, the zero-new-id
result is misclassified as genuine exhaustion, the loop breaks `Ok`, and the third
sibling `C(T)` is SILENTLY LOST.

The verifier's prescribed fix: in the zero-new-id branch, when
`new_ids == 0 && returned >= cap`, also detect the stall on the FIRST re-request
by checking whether `page_back(returned, cap, oldest) == Some(current_until)` —
i.e. page_back would re-pin the SAME `until`. If so, the relay is already pinned on
this very first re-request and the stall is certain → return the requeue
`Err(RelayError::FetchTimeout(...))` immediately. This removes the `prev_until`
dependency for stall detection and catches the no-newer-event case. The genuine-
exhaustion break is preserved for `returned < cap` and for the case where page_back
would actually advance `until` to an older second.

Purpose: restore the RELAY-03 / project core completeness invariant ("each follow
list fetched completely; never silently drop pubkeys") at the cap boundary when
every event shares the oldest second — the exact path 02-10 left open.

Output: a strengthened stall-detection condition in `paginate_chunk` plus a
companion test in `tests/pagination.rs` that seeds a pool with no newer event,
FAILS before the fix (silent truncated Ok), and PASSES after (Err surfaced) — while
the existing `deterministic_boundary_stall_surfaces_error` (2-visit path) stays
green.
</objective>
